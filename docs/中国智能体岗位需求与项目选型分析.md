# 最适合我的 2 个项目方向

> 更新时间：2026-04-22  
> 目标：基于中国市场真实招聘需求、你当前的 Python / AI 应用后端背景、以及你已经重构好的 LangGraph 单智能体后端，选出最适合你立刻执行的 2 个项目，并拆成可以直接开发的方案。

## 1. 结论总览
**第 1 名：可执行的售后客服工单智能体**  
**第 2 名：客服坐席 Copilot**

这是最适合你的两条线，不是因为它们“听起来热门”，而是因为它们同时满足这 4 个条件：

- 和中国近 30 天公开招聘里的重复需求模式最贴近
- 和你当前 `Python + LangGraph + 单智能体业务后端` 的能力最匹配
- 最容易做出完整业务闭环，而不是停在 demo
- 最容易在简历和面试里讲清楚 ROI、系统设计和落地价值

我不建议你此时优先做：

- 通用聊天产品
- 多智能体平台
- 通用 Agent marketplace
- 纯 RAG 知识库助手
- 过于泛的“企业内部全能 Copilot”

## 2. 为什么选这 2 个
### 2.1 市场判断
中国市场近 1 个月公开招聘里，重复出现最多的不是“通用平台能力”，而是下面这类组合：

- `Python / FastAPI`
- `AI应用开发工程师 / Python后端（AI Agent方向）`
- `智能客服`
- `工单系统`
- `知识库`
- `RAG + Agent`
- `工作流 / 多步执行`
- `智能辅助人工`
- `稳定性 / 权限 / 日志 / 审计`

### 2.2 为什么是这 2 个，而不是别的
**为什么不是商家经营 / 销售运营助手排前两名**

- 这个方向有需求，但更偏“经营分析 + 策略推荐 + 运营动作”，比客服场景更抽象。
- 对求职第一枪来说，ROI 叙事没有客服/工单那么直观。
- 更适合作为第 3 个差异化项目，而不是第 1、第 2 个核心项目。

**为什么不是企业内部员工服务助手**

- 真需求有，但公开招聘信号弱于客服和工单。
- 内部服务类场景更依赖真实企业权限、组织架构、审批关系和系统上下文。
- 没有真实企业接口时，做出来更容易显得“像流程图，不像产品”。

**为什么不是带审批的业务操作 Copilot**

- 这是能力形态，不是明确项目。
- 它应该作为你项目的底层特征存在，而不是单独包装成项目名。

**为什么不是垂直行业“服务+办理”助手**

- 方向没错，但过宽。
- 对你当前阶段，先把客服/工单这个高频招聘方向做透，比做泛垂直更稳。

### 2.3 这 2 个项目分别更贴哪些岗位
**项目一：可执行的售后客服工单智能体**

- `AI应用开发工程师`
- `Python后端（AI Agent方向）`
- `智能客服研发`
- `客服平台 / 工单系统 / 企业服务 AI`

**项目二：客服坐席 Copilot**

- `智能客服研发`
- `AI应用工程师`
- `客服智能化 / 客服效能 / 知识库 / 质检相关 AI`
- `企业服务 AI 工程师`

### 2.4 对求职竞争力的帮助点
这两个项目组合起来，刚好覆盖招聘市场最常见的两类 AI 能力：

- **项目一证明你会做“执行型 agent”**
  - 查数据
  - 调工具
  - 发起动作
  - 审批
  - 回执
- **项目二证明你会做“辅助型 Copilot”**
  - 总结
  - 推荐
  - 检索
  - 补全
  - 风险提示

这比只做一个通用聊天 bot 强很多，也比一上来做平台更贴近招聘需求。

## 3. 项目一：可执行的售后客服工单智能体
### 3.1 项目定位
一个面向电商 / 本地生活 / 出海 SaaS 的售后执行型智能体后端。  
它不是只回答问题，而是能围绕售后流程完成“查 + 办 + 审批 + 回执”。

### 3.2 用户与买单方
**目标用户**

- 电商客服
- 售后专员
- 客服主管
- 平台服务团队

**买单方**

- 电商平台
- 商家服务商
- 本地生活平台
- SaaS 客服系统客户

### 3.3 业务问题
这个项目解决的是客服和售后流程里的高频、可结构化动作：

- 用户咨询订单和物流
- 用户追问退款和补偿进度
- 客服需要创建工单、升级工单
- 客服需要在规则约束下发起退款、补偿、改地址
- 高风险动作需要主管审批
- 每一步都需要可追踪、可回执

### 3.4 MVP 范围
第一阶段只做 5 个闭环，不扩散：

- 查订单状态
- 查物流状态
- 查退款进度
- 创建 / 升级售后工单
- 发起退款 / 补偿申请并审批

**第一阶段必须做**

- 对话入口
- tool calling
- 工具级审批
- 执行结果回执
- 审计日志
- 最小工作台

**第二阶段再补**

- 多轮售后策略优化
- 更复杂的退款规则
- 多渠道接入
- 质检和报表
- 多租户和细粒度权限

### 3.5 用户流程
**核心流程 1：查单 + 查物流**

1. 客服输入“帮我看下订单 A10086 现在到哪了”
2. 智能体调用订单和物流工具
3. 返回订单状态、物流节点、建议回复
4. 记录本次工具调用和会话日志

**核心流程 2：创建工单 + 升级**

1. 客服输入“这个用户情绪很差，帮我建售后工单并升级”
2. 智能体读取订单上下文和客户信息
3. 自动创建工单，填写分类、优先级、摘要
4. 如升级为高优先级，则要求主管审批
5. 审批通过后执行升级并回执

**核心流程 3：退款 / 补偿**

1. 客服输入“用户要退款 200 元，原因是商品破损”
2. 智能体查询订单、退款规则、历史申请
3. 生成退款申请或补偿建议
4. 命中高风险规则时进入审批
5. 审批后继续执行，返回申请单号和结果

- 查询类：直接执行
- 普通建单：直接执行
- 升级高优工单：审批
- 退款：审批
- 补偿：审批
- 改地址：审批

**是否需要审计**

**最先应该实现的 3 个功能**

1. `get_order + get_shipment`
2. `create_ticket`

**MVP 验收场景**

- 能查单并给出物流回执
- 能创建工单并显示工单编号
- 退款动作一定能触发审批并在审批后继续执行

## 4. 项目二：客服坐席 Copilot
### 4.1 项目定位
一个面向人工客服的辅助型智能体。  
它不替代人执行售后动作，而是帮助坐席更快、更准地处理会话、补全工单、检索 SOP、生成回复建议。

### 4.2 用户与买单方
**目标用户**

- 一线客服坐席
- 客服主管
- 质检团队
- 培训团队

**买单方**

- 电商客服中心
- BPO 公司
- 客服系统服务商
- 企业服务团队

### 4.3 业务问题
这个项目解决的是“人工客服效率低、质量不稳定”的问题：

- 客服需要手动总结会话
- 客服需要自己检索 SOP 和知识库
- 工单字段经常漏填
- 回复质量依赖个人经验
- 高风险会话识别慢
- 质检和培训成本高

### 4.4 MVP 范围
第一阶段只做 5 件事：

- 自动总结当前会话
- 生成回复草稿
- 检索 SOP / FAQ / 知识库
- 自动补全工单字段
- 给出风险标签和升级建议

**第一阶段必须做**

- 会话摘要
- 回复建议
- 知识检索
- 工单补全
- 风险标签

**第二阶段再补**

- 自动质检评分
- 多轮训练 / 质检复盘
- 多模态语音转写
- 坐席绩效分析
- 多渠道坐席统一面板

### 4.5 用户流程
**核心流程 1：总结 + 推荐回复**

1. 坐席打开用户会话
2. Copilot 自动读取最近消息窗口
3. 生成会话总结和当前问题归因
4. 给出 2-3 条回复草稿
5. 坐席手动编辑后发送

**核心流程 2：SOP 检索 + 工单补全**

1. 会话涉及特定售后规则
2. Copilot 检索知识库和 SOP
3. 输出适用规则和推荐处理动作
4. 自动补全工单标题、分类、优先级、摘要

**核心流程 3：风险识别**

1. Copilot 检测到情绪升级或高风险售后诉求
2. 标记风险标签
3. 建议升级到主管或售后专员
4. 可一键创建升级任务

### 4.6 需要的 tools / API / 审批 / 权限 / 审计 / RAG
**需要的 tools**

- `get_conversation_window`
- `get_customer_profile`
- `get_ticket_context`
- `search_sop`
- `search_kb`
- `draft_ticket_fields`
- `flag_risk`
- `create_escalation_task`

- 不是核心

- 第一版不需要把 approval 做成主流程
- 如果你想复用现有 approval 能力，只在“升级高优先级工单”这类动作里使用

**是否需要权限**

- 需要

  - `agent`

  - `qa_manager`

**是否需要审计**

- 需要

### 4.9 开发顺序

**最先应该实现的 3 个功能**

**最容易拖慢开发的坑**

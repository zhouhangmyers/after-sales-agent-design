# 手记全流程笔记-骨架版

## 当前范围

- 当前骨架只覆盖 `src/agent_service/config.py`
- 当前骨架只覆盖 `src/agent_service/main.py`
- 当前目标：沉淀启动链路里的主线、顺序、因果、工程模式

---

## 一条总主线

`Settings -> create_app -> lifespan.startup -> bootstrap_app_state -> close_app_state`

展开后是：

`配置读取与裁决 -> 应用外壳创建 -> 生命周期切入 -> 运行期依赖装配 -> 统一资源回收`

---

## config.py 骨架

### Settings 的工程角色

- 启动配置对象
- 配置裁决入口
- 启动前约束关口

### 配置链路

`外部配置输入 -> BaseSettings / model_config 读取 -> 字段类型化 -> 派生属性 -> model_validator 组合校验`

### 字段分层

- 应用基础配置
- 数据库与持久化配置
- 安全与跨域配置
- LLM 提供商配置
- Agent 运行时配置
- 并发与事件总线配置

### 高复利模式

- `@property`：派生配置，不重复把判断逻辑散落到业务代码里
- `@model_validator(mode="after")`：启动前组合校验，不只校验单个字段类型，还校验整组配置是否能真实跑起来

### 当前关键派生结果

- `is_production`：生产环境判定结果
- `parsed_cors_allowed_origins`：CORS 最终来源列表
- `resolved_run_event_bus_backend`：事件总线最终后端

### 当前关键约束分类

#### 运行拓扑约束

- 多 worker 不能配 SQLite
- `postgres_notify` 依赖 PostgreSQL
- 多 worker 不能使用 `inprocess` 事件总线

#### 生产环境约束

- 生产环境不能使用 SQLite
- 生产环境不能自动建表
- 生产环境必须配置 `LANGGRAPH_POSTGRES_URL`
- 生产环境必须配置 `API_KEY`
- 生产环境必须配置明确的 CORS 白名单
- `deepseek` 在生产环境下必须有 `DEEPSEEK_API_KEY`

### config.py 的抽象结论

- `Settings` 不是普通参数表
- `Settings` 是启动前的统一配置裁决对象
- 配置读取、派生、约束收口在同一个入口里

---

## main.py 骨架

### 总链路

`Settings -> create_app -> lifespan.startup -> bootstrap_app_state -> close_app_state`

### bootstrap_app_state 内部链路

`db_manager -> run_event_bus -> resolved factories -> tool_registry -> workflow_registry -> langgraph_checkpointer -> workflow_service`

### shutdown 链路

`workflow_service -> langgraph_checkpointer -> run_event_bus -> db_manager`

---

## create_app 骨架

### 工程角色

- 应用启动总入口
- 应用外壳装配点
- 生命周期接入点

### 结构顺序

1. resolve 配置与默认工厂
2. 初始化日志系统
3. 定义 `lifespan`
4. 创建 FastAPI 应用实例
5. 注册异常处理器、中间件、路由

### 当前可扩展装配位

- `tool_executor_factory`
- `workflow_registry_factory`
- `workflow_service_factory`

### 当前模式

`外部注入工厂 or 默认工厂`

### 当前抽象结论

- `create_app()` 先搭应用外壳
- `create_app()` 不在函数体里一次性完成全部运行资源创建
- 默认工厂 + 注入覆盖是平台化扩展点

---

## lifespan 骨架

### 工程角色

- startup 装配入口
- shutdown 回收入口

### 当前链路

- startup：调用 `bootstrap_app_state()`
- shutdown：调用 `close_app_state()`

### 当前结论

- 应用外壳创建和运行资源启动不是同一时刻
- 应用对象先创建
- 共享资源在生命周期阶段正式进入运行态

---

## bootstrap_app_state 骨架

### 工程角色

- 运行期共享依赖装配中心
- `app.state` 挂载入口

### 依赖顺序

#### 1. `db_manager`

- 基础持久化资源
- 后续运行资源的底层依赖之一

#### 2. `run_event_bus`

- 运行事件实时通知通道
- 当前挂载的是 handle，不是裸对象

#### 3. resolved factories

- `resolved_tool_executor_factory`
- `resolved_workflow_registry_factory`
- `resolved_workflow_service_factory`

#### 4. `tool_registry`

- 基于 `ToolExecutor` 生成
- 对外暴露的是工具注册结果，不是裸 `ToolExecutor`

#### 5. `workflow_registry`

- 基于 `ToolRegistry` 生成
- 定义系统启动后可用的 workflow definition 集合

#### 6. `langgraph_checkpointer`

- 工作流状态持久化资源
- 当前挂载的是 handle，不是单独 saver

#### 7. `workflow_service`

- 依赖 `settings`
- 依赖 `tool_registry`
- 依赖 `checkpointer`
- 核心运行服务放在依赖都到位之后再装配

### 当前结论

- 装配顺序不是随意排列
- 先基础设施
- 再注册中心
- 最后核心服务

---

## close_app_state 骨架

### 工程角色

- startup 装配的对称回收入口
- 应用 shutdown 阶段的统一资源回收点

### 回收顺序

1. `workflow_service.close()`
2. `langgraph_checkpointer.close()`
3. `run_event_bus.close()`
4. `db_manager.dispose()`

### 回收链路

`workflow_service.close() -> tool_registry.close() -> ToolExecutor.close() -> tool runtime.close()`

### 当前回收对象

- 工作流服务资源
- LangGraph checkpoint 资源
- 事件总线资源
- 数据库管理器与连接池资源
- tool runtime 背后的 worker / 队列等运行资源

### 当前结论

- shutdown 基本遵循后创建的先释放
- 上层运行服务先停
- 底层数据库资源最后释放

---

## 高复利工程模式

- 配置对象不只是参数表，而是启动裁决入口
- 启动入口先搭应用外壳，真正资源装配在生命周期里发生
- 共享依赖统一挂载到 `app.state`
- 默认工厂 + 注入覆盖是平台化扩展点
- handle 模式表示“资源对象 + close 动作”
- shutdown 基本遵循“后创建的先释放”

---

## 当前阶段最短主线

`配置 -> 外壳 -> 生命周期 -> 基础设施 -> 注册中心 -> 核心服务 -> 资源回收`

`Settings -> create_app -> lifespan -> bootstrap_app_state -> close_app_state`

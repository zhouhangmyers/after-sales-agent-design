#### 1. 确定本次启动实际采用的 settings 和默认工厂


#### 1.1 可扩展装配位：默认工厂 + 注入覆盖

这一块是很值得长期记的工程模式：

```py
# 外部可以传工厂覆盖默认装配逻辑
# 如果外部没传，就走平台内置默认实现
resolved_xxx_factory = 外部传入工厂 or 默认工厂
```

它的复利点不在于几个具体函数名，

而在于它体现了一个很通用的系统装配规律：

- 核心对象的创建逻辑不要完全写死
- 默认路径要能直接跑起来
- 同时要给测试、扩展、替换实现预留注入点

在这个项目里，这 3 个装配位分别控制：

- `tool_executor_factory`：底层工具执行器怎么建
- `workflow_registry_factory`：系统启动后有哪些 workflow definition 可用
- `workflow_service_factory`：核心工作流服务怎么按依赖组装出来

所以这里真正值得背的不是函数名本身，

而是这句抽象话：

`启动装配里，底层执行器、工作流注册表、工作流服务都支持外部注入；如果没注入，就走默认工厂。`

#### 3. 创建 lifespan 生命周期管理器

```py
@asynccontextmanager
async def lifespan(app: FastAPI):
    logger.info("startup")
    # 这里是应用起始阶段，将settings 和 工厂 来初始化 app 要用的共享依赖
    bootstrap_app_state(
        app,
        resolved_settings,
        tool_executor_factory = resolved_tool_executor_factory,
        workflow_registry_factory = resolved_workflow_registry_factory,
        workflow_service_factory = resolved_workflow_service_factory,
    )
    try:
        yield
    finally:
        logger.info("shutting down")
        # 这里是应用结束阶段，用来自动关闭 app 的共享依赖，防止资源泄露
        close_app_state(app)
```

### 文件函数(close_app_state)

```py
def close_app_state(app: FastAPI) -> None:
    app.state.workflow_service.close()
    app.state.langgraph_checkpointer.close()
    app.state.run_event_bus.close()
    app.state.db_manager.dispose()
```

#### 1. 这是 shutdown 阶段的统一资源回收入口

```py
# create_app() 里定义的 lifespan 在应用结束时会进入 finally
# finally 里调用 close_app_state(app)
# 所以这里就是整个应用 shutdown 时统一清理共享资源的地方
```

#### 2. 回收顺序不是随便写的，而是基本遵循“后创建的先关闭”

```py
# startup 装配时的大致顺序：
# db_manager
# run_event_bus
# tool_registry
# langgraph_checkpointer
# workflow_service

# shutdown 回收时就按反方向来：
# workflow_service
# langgraph_checkpointer
# run_event_bus
# db_manager
```

#### 3. 第一层先关 WorkflowService

```py
# 这里不能简单理解成“关一个 service 对象”
# WorkflowService.close() 继续往里会做两件事：
# 1. self._tool_registry.close()
# 2. self._llm_service.close()

# 而 ToolRegistry.close() 又会继续关 backend_executor
# backend_executor.close() 最终会走到 tool runtime.close()

# tool runtime.close() 内部会强制停掉 worker，并回收队列资源
# 所以这里回收的不只是业务服务本身，
# 还包括工具执行链路背后的并发运行资源
```

这里你可以这样理解：

- `workflow_service.close()` 是最上层业务运行资源的回收入口
- 它会顺带把内部持有的 `tool_registry`、`llm_service`、`ToolExecutor`、`tool runtime` 一路关下去
- `tool runtime` 里不只是普通对象，还涉及 worker 子进程、请求队列、响应队列这些执行期资源

所以这一层最应该先停，因为它最靠近“业务运行态”。

#### 4. 第二层关闭 LangGraph checkpointer

```py
# checkpointer 是工作流运行过程里的状态持久化资源
# startup 时挂的是 LangGraphCheckpointerHandle
# 里面同时带 saver 和 close
# shutdown 时调用的是这个 handle.close()
```

这里的重点是：

- 先把上层 `workflow_service` 停掉
- 再关状态保存器
- 这样可以避免还在跑的工作流继续访问已经被关闭的 checkpoint 资源

#### 5. 第三层关闭 run_event_bus

```py
# run_event_bus 负责运行事件的实时通知
# 它也是一个 handle，里面带 bus 和 close
# shutdown 时通过 close() 释放订阅、监听或进程内广播相关资源
```

这里关闭的是“实时事件传播通道”，不是数据库本身。

#### 6. 第四层最后释放数据库管理器

```py
# db_manager.dispose() 最后执行
# 它对应的是数据库连接池、engine 等底层持久化资源释放
```

数据库资源放到最后释放的原因是：

- 上层 service、checkpointer、event bus 在关闭过程中都可能还需要用到底层数据库能力
- 所以数据库不能抢先关
- 必须等上层都停稳之后，再统一 dispose

#### 7. 这段 shutdown 的本质

可以把这段总结成一句话：

`close_app_state()` 是 startup 装配的对称回收入口，它在应用结束时按“业务运行层 -> 状态持久化层 -> 事件传播层 -> 数据库基础设施层”的顺序统一回收共享资源，避免资源泄漏和关闭顺序错误。

## 知识要点核心：

- `create_app()` 是应用总装配入口；真正 startup 装配发生在 `lifespan -> bootstrap_app_state()`
- `Callable[...]` 表示“工厂函数类型提示”：`Callable[[], ToolExecutor]` 就是“传一个零参数函数，返回 ToolExecutor”
- `lambda` 是最短写法； lambda:ToolExecutor(...) 表示一个0参函数返回ToolExecutor实例对象
- 这套 `xxx_factory` 参数属于平台化冷启动注入点；外部最常传的是 `tool_executor_factory`，后两个更多是可扩展装配位
- `app.state.xxx` 的 `xxx` 不是 FastAPI 固定字段，而是项目自己定义的应用级共享依赖挂载名
- `ToolExecutor` 是底层工具执行器，内部核心参数是：`tools`、`tool_policies`、`timeout_seconds`、`runtime`
- `ToolRegistry.from_tool_executor(...)` 会基于 `ToolExecutor` 整理出 `tools / tool_policies / tool_timeout_seconds / backend_executor`
- `ToolExecutor` 不单独挂到 `app.state`，因为它已经被封装进 `ToolRegistry.backend_executor`；请求期真正对外暴露的是 `tool_registry`
- `workflow_registry_factory(tool_registry)` 返回的是 `WorkflowRegistry` 实例，不是类定义；它内部装的是一组 `WorkflowDefinition`
- 默认 `build_default_workflow_registry(tool_registry)` 会把 `tool_registry.tool_names` 填到 workflow 的 `allowed_tools`
- `tool registry is empty` 的 warning 关键条件是 `tool_registry.tool_names` 为空；默认 `ToolExecutor(timeout_seconds = ...)` 不会自动携带工具列表
- `run_event_bus` 的作用是做 run 事件实时通知：数据库负责落库和回放，event bus 负责实时推送
- `run_event_bus` 和 `langgraph_checkpointer` 挂到 `app.state` 的都是 handle；里面同时带“真正资源对象 + close 关闭动作”
- run_event_bus 不负责往三张表落库，它负责把“已经落到 run_events 的新事件”实时广播出去；SSE 输出的本体是 run 事件流，历史部分来自 RunEventRecord，实时部分来自 PublishedRunEvent，而不是 RunRecord。
- `close_app_state()` 是 startup 装配的对称回收：关闭 `workflow_service`、`checkpointer`、`run_event_bus`，最后释放数据库资源
- `workflow_service.close()` 回收的不只是 service 自己，它还会继续关闭 `tool_registry`、`llm_service`、`ToolExecutor` 和底层 `tool runtime`
- `tool runtime` 的 close 涉及 worker 子进程和队列资源回收，所以这里更准确地说是“运行期并发执行资源回收”，不只是简单的对象销毁
-
